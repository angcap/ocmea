Overview - #back to top

The solution proposed for the online house design application at Factory Home is a Java EE application based on an N-tier architecture. N-tier application architecture provides a model by which the solution is flexible and software components are more testable and reusable. The architecture proposed comprise a Presentation Layer with views and presentation templates which generate the interaction interface for the application, a Business Layer where business rules and logic resides, an Integration Layer containing the implementation for persistent data and external systems access, a Domain Layer containing objects used to model data. The Java EE 7 platform is used for it's wide range of technologies supported and the easy of development in the context of enterprise applications.

Key Assumptions - #back to top

  1. I've have expanded the use case diagram provided in the assignment in order to reason about and show critical technical aspects of design and implementation of SuD such:
    • its integration with enterprise ecosystem
    • authentication and authorization
    • detailed definition of system behaviour required to plan the implementation of system features taking into account their dependencies
  2. The requirements document states that "The company plans to launch the System under Development (SuD) with a limited number of valid combinations for house styles and accessories to test the market", so
    • I assume that the managing of what house styles are available to the SuD and what accessories are active for every style, is entirely accomplished by the stock inventory and management system. The SuD uses the stock inventory and management system provided web services to obtain such information and drive the user from the beginning of house design toward applicable components for current design style.
  3. With House Style i refer to one of the styles known to the inventory and stock management system enabled for the SuD, an object representing such style is a part of an House Design which, with that particular style and status, is referred to a particular user and represent the whole design of the house.
  4. I assume that only registered users can access the SuD, than the SuD will implement an authentication and authorization mechanism.
  5. The requirements document states that "The company considers that providing strong visual feedback to end users on how their dream home looks is of great priority. To support this requirement, Factory Homes has also invested in a 3-D modeling tool which, given an XML representation of a house, can output high quality models of the house that can be rendered to an end user as images, video, or PDF", therefore
    • I assume that users can choose only one of the possible rendering formats as the preferred one's at registration time and eventually modify this preference later before they complete design.
    • I haven't much information about such 3-D modeling tool, it could be a command line executable program, located on a dedicated machine, that take an xml file and a format type as input and generate the rendering file on a local filesystem; or it could be a cloud service API.
      Anyhow, i assume that the generation of the 3-D model is an expensive operation and that the SuD is decoupled from the 3-D modeling tool by a SOAP based web service, i consider in the scope of the SuD only the interaction with such (preexisting or out of the scope) web service.
      Then the SuD implements the software components needed to generate the xml structure of the design and use such xml as payload for the 3-D modeling system web service request.
  6. The SuD will implement a "my designs section" from which user can find all designs, view/change their status, view the rendering or mark the design complete, add components to the design, create a new one.

Key Decisions -#back to top

  1. The solution presented leverage the followings J2EE platform technologies:
    • JPA: i have choosen it because help to enforce a structured object oriented approach in a database neutral manner, leaving perstistence as a secondary concern.
    • Enterprise Java Beans, i have choosen both Stateless and MessageDriven because both synchronous and asynchronous business logic is required and because EJB provide a wide range of almost transparent infrastructure facilities such transaction management, connection pooling, security management, remoting, concurrency, scheduling.
    • JAAS: it allows to easely and robustly integrate enterprise security solutions and framework in the application using standard API.
    • CDI: it makes easy for developers to use enterprise beans along with JavaServer Faces technology in web applications, it gives a great deal of flexibility to integrate various kinds of components in a loosely coupled but typesafe way.
  2. There are 2 kind of users interacting with SuD: internet users that access the system in order to create a design and field sales representatives that access the system in order to find completed designs and schedule appointments with customers.
    I plan to implement 2 kind of access methods, both leveraging the JAAS technology which, in a declarative manner, prevent unauthenticated access to the SuD and prevent access to section of the SuD to unauthorized users:
    • one access method for online house designers that faciltate use of the SuD by using a public identity to identify users, i.e. the user performs a social login (with facebook or google or twitter) and then, if not yet registered in the SuD database, the information obtained by identity provider are eventually integrated with data requested to the user with a precompiled form.
      This method require an implementation to perform a JAAS login and i've choosen the SocialAuth java library to support the development of this feature in order to reduce the effort required and to support a wide range of social networks identity, furthermore this library could be used in future features to give customers the ability to share their designs on social networks in order to expand the company's visibility and attract more customers;
    • one access method for field sales representatives that are requested to enter username and password to access the SuD; this method use the J2EE's compliant application server capabilities to use a database realm in order to perfom a JAAS login.
  3. Since i assumed that the generation of the 3-D model is an expensive operation,
    i have choosen an asynchronous pattern with the JMS technology to implement a reliable and scalable asynchronous elaboration of the generation of the 3-D model.
    • This decision has an implication for the Complete House Design use case, in fact, when the user marks an house design as completed in the SuD, the system will respond with a message indicating that the actual completion of the generation of the rendering will be notified to the user with an email within a configurable amount of time (for example 15 minutes).
      This is behaviour is implemented by putting the request for the generation of the 3-D rendering in a JMS queue and let an MDB to elaborate it and, upon generation is completed, also put the request for the email notification generation and delivery into another JMS queue and let another MDB to elaborate it.
      Rather than keep the user waiting for the generation and have a bottleneck in the overall architecture with many concurrent requests for an expensive operations flooding another program or system, this decision gives a great element of control and scalability (tuning JMS throttling we can keep system load under a costant factor) and flexibility (configuration and decoupling between software components to the SuD.
  4. The data layer is based on the JPA technology, indeed domain model is implemented with JPA entities that also encaplsulate business data rules. This allow a natural integration with EJB centered business layer and give the flexibility to use JPA annotated POJOs as business domain objects in initail development and/or for simpler use cases.
  5. The Contexts and Dependency Injection is used throughout the SuD to integrate various kinds of components in a loosely coupled but typesafe way.

Class Diagram - #back to top

Notes

  • The business domain model concept of CompletedDesign is modeled with the more general concept of design by the HouseDesign class which has a persistent attribute which indicate the status of design (draft at creation, completed and accepted) and two business methods to change it.
  • The HouseStyle class act as container for the reference to the house style of a design, it holds one of the enabled styles for online house design application.
  • I assume that for every design the system adds all the RequiredFeature configured in the relational database corresponding table.
  • Annotations have been used to show:
    • how specific items in the domain model have been mapped onto Java EE components like Entity classes and session or message driven beans
    • how CDI Qualifiers are used to apply the Dependency Inversion Principle
  • I assumed that web services exposed by inventory and stock management system are two distinct SOAP based endpoints, one for stock management and the other for inventory management.
  • I assumed that the interaction with the 3D Modeling System is accomplished by invoking a SOAP based web service endpoint.
  • A combination of the Proxy Pattern (GoF) and the Gateway Pattern (Fowler) is used to handle the interaction with inventory and stock management system, InventoryPortType and StockPortType are jaxb generated web service clients.
  • A combination of the Proxy Pattern (GoF) and the Gateway Pattern (Fowler) is used to handle the interaction with modeling system, ModelingPortType is a jaxb generated web service client.
  • HouseDesignFacade implementation rely upon the @InventoryServiceProxy implementation of the InventoryService interface to get all available HouseStyles and then use the HouseStyle as a key parameter on subsequent interactions in order to get only applicable components: this can be accomplished either propagating the HouseStyle to the Inventory System (if this is the behaviour of IS web service) or by implementing this behaviour as business rule in the implementation of InventoryService interface.

Component Diagram - #back to top

Note

Business and Integration layers are decomposed in interfaces and implementations following the Dependency inversion principle.
The Domain Layer contains the JPA Entities and POJOs used as Transfer Objects. Since JPA Entities are annotated POJOs, they can be used across all layers once detached from persistent context with the constraint that relations and dependent objects can not be retrieved out of that context: to overcome this constraint appropriate Transfer Objects needs to be added and used in Business Layer.

Deployment Diagram#back to top

Notes

The proposed solution for deployment shows 3 tiers: Presentation, Business, Persistence and Integration.
Network is segmented following security best practices to enforce strict firewall configuration in order to enable only needed communication paths among tiers.
SSL is used to ensure secure communication channel between user web browser and application, secure channel ends at load balancer boundary in order to not affect performance of application servers.
Even if not explicitly stated as clear NFR, proposed solution considers mantaining stand-by server to address Failover and symmetric clustering to provide high availability and balancing. Balancing at front end is addressed with a dedicated appliance or software component, while JEE application server's cluster capabilities address clustering and failover for EJB calls between front end and business tiers.
Different hardware profile are depicted with stereotype HW profile Xin order to show different characteristics of machines in every tier:

Proposed solution could also be easely implemented in cloud environment, especially on specific java oriented platform such as Oracle Java Cloud Service which supports JEE 7, provide Weblogic application server runtime environment, Coherence cache and database server with all high availability options.

Top 3 Technical Risks#back to top

  • Security is a critical risk factor for an Internet connected system.
    I have mitigated security related risk for SuD with:
    • network segmentation (see deployment diagram) and use of HTTPS to prevent Man-in-the-Middle attacks
    • use of authentication and authorization to limit access to the system
    • use of JPA with query parameter binding to reduce risk of SQL injection
  • Performance is a critical risk when interaction with external systems is embedded in core features, especially when at one of the ends of the interaction there are users.
    I have mitigated performance related risks due to interaction with external system with the introduction of a cache at the integration layer, the JCache API can be used as a drop-in addition in the Java EE application then, the InventoryProxy's methods that interact with Stock Inventory and Management System to retrieve data with a low mutability rate can be annotated with the CacheResult annotation. In this way the caching is plugged-in almost transparently leaving the concrete aspects to be addressed with deployment descriptors and server configuration.
  • Availability is a very critical point for an internet web site supporting a core business, the company can be perceived as unreliable and loose potential customers if site is not available.
    I have mitigated availability related risks designing for high availability and reliability, using clustering for critical components and providing failover nodes.
